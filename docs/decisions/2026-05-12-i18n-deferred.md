# i18n deferred — CN-first product (P26)

**Date:** 2026-05-12
**Status:** declined (revisit on trigger)

## Context

Chinese strings are hardcoded throughout the codebase:

- DTO field-level validation messages: `@IsString({ message: 'account_id必须是字符串' })`
- Service-layer business errors: `throw new Error('账号不存在，请检查')`
- Filter error messages: `'缺少签名参数'`
- Entity / column metadata: `@Column({ comment: '用户基础信息表' })`

Total surface area: ~100+ strings across access / basic / oplog DTOs, the auth / sms / log services, the four exception filters, and a long tail of entity comments.

## Considered options

### A. Full i18n migration (`nestjs-i18n`)

Install `nestjs-i18n`, add an `Accept-Language` resolver, externalize every Chinese string into `i18n/{zh-CN,en}/*.json`, replace every literal with a `t('key.path')` call. Multi-PR effort, churns every DTO file.

### B. Infrastructure-only + filter layer

Wire up `nestjs-i18n`, translate the dozen filter-layer messages and the most user-facing service errors (login / auth), leave DTOs as a documented follow-up. Single PR, demonstrable but partial.

### C. Decline — document the position

This file. Don't touch the code.

## Decision

**Option C.** Decline i18n work for now.

### Reasons

1. **Audience.** The product is CN-first. Every current consumer reads Chinese. There is no concrete English-language client on the roadmap.
2. **Cost / value asymmetry.** Option A is a multi-week churn touching every controller / DTO. Option B is half-built infrastructure that doesn't actually serve any user until A also lands.
3. **Premature abstraction risk.** A translation key scheme designed without an English customer's input is almost always rewritten when one shows up — different domain vocabularies, different formality registers, different field-grouping conventions.
4. **Real cost today is small.** The hardcoded strings are visible in error responses but those responses already have a stable `code` field; clients that care can switch on the code, not the message.

## Revisit trigger

This decision **must** be revisited when any of the following becomes true:

- An English-language customer / partner is named for delivery within 90 days.
- The codebase grows a second language requirement (e.g. JA, KO) — at that point an actual i18n abstraction starts paying for itself.
- A regulator or contract requires localized error messages (rare, but happens in fintech / health verticals).
- Any new module is being built that already has multiple-language requirements from day one — wire `nestjs-i18n` then for that module only, don't retrofit existing code.

When the trigger fires, the order of work is:

1. `nestjs-i18n` + `Accept-Language` resolver + zh-CN/en translation files.
2. Filter layer (~10 messages) — highest leverage, smallest surface.
3. Auth + tenant control-plane services — second priority, customer-facing.
4. Per-module DTO field messages — long tail, can be migrated module-by-module.
5. Entity column comments — last; pure documentation, no runtime effect.

## Adjacent debts

These remain real and addressable independently of i18n:

- DTO field messages occasionally don't match the field name (typo / copy-paste drift). A targeted lint rule (`'must contain field name'`) would catch the most common authoring mistake.
- Service-layer `throw new Error('...')` mixes business errors with system errors. Migrating those to typed `BusinessException` / `HttpException` subclasses would let filters key on type rather than parsing strings — and would make a future i18n migration mostly mechanical.
